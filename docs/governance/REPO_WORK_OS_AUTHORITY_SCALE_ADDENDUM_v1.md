# Repo Work OS — Authority Scale Addendum v1

Effective: 2026-07-24

Status: Generalized doctrine addendum produced from the five-repo authority-scale migration. This file is an addendum artifact; it does not claim to replace or silently rewrite any unavailable canonical Repo Work OS source document.

## 1. Generated Authority Scale Law

When a repository is intended to pursue material SEO/AEO/GEO, citation, reference, or LLM-surfacing growth, the implementation must support maximum legitimate fanout across the dimensions that are actually relevant to that domain.

- A 100,000-opportunity materialized runway is a minimum scale benchmark for large authority programs, not a ceiling.
- Opportunity count is never a page quota.
- The system must support semantic clustering, duplicate suppression, entity ownership, page-strategy admission, and risk-adjusted source/claim gates before route creation.
- Fanout may be larger than the materialized corpus through deterministic expansion grammar, entity taxonomies, compressed shards, or on-demand materialization.
- Repository byte size is not a fanout KPI.

## 2. Accepted Output Freeze Law

For generated authority content expected to scale materially, accepted public content must not silently mutate during ordinary builds.

Lifecycle:

`discover -> admit -> generate/repair -> validate -> accept -> freeze`

Existing accepted content changes only through:

`exact scope -> thaw -> mutate -> validate -> refreeze`

Requirements:

- Freeze the authoritative content/output layer appropriate to the repo.
- Dynamic application state, user state, checkout, admin, authentication, telemetry, and other runtime state must not be frozen merely because the repo also contains authority content.
- Global shell changes may be handled separately from frozen semantic/page content when architecture permits.
- Freeze storage should be compressed/content-addressed when practical.
- A cache may improve performance, but validation cache is not source of truth. Accepted-output freeze is a correctness boundary, not merely a performance cache.

## 3. Intelligence Storage Law

Maximum fanout capability must not require maximum repository bloat.

Preferred hierarchy:

1. compact canonical seeds/entities/dimensions/expansion rules;
2. compressed or sharded observed/planning intelligence;
3. small action/admission queues;
4. compressed content-addressed accepted-output freeze;
5. manifests, hashes, and receipts sufficient for deterministic validation and rollback.

Avoid unnecessary simultaneous duplication of canonical, staged, published, rendered, frozen, and temporary forms unless each representation has a distinct authoritative purpose.

## 4. Dynamic Velocity Law

Publication numbers are safety ceilings, not quotas.

A repo may scale publication upward only when evidence supports it, including:

- sufficient semantically distinct admitted backlog;
- appropriate source and claim coverage;
- clean validation;
- acceptable duplicate/thin-content rates;
- stable indexation/distribution capacity;
- observed surfacing/citation yield where measurable.

The system must scale down, skip, repair, or pause a lane when those conditions deteriorate. High fanout capacity never authorizes unsafe publication volume.

## 5. 100K Surfacing Acceleration Law

For repos governed by a 100K citation/reference/LLM-surfacing objective:

- maximize legitimate query, entity, intent, persona, decision, format, geographic, freshness, and conversational/LLM fanout where relevant;
- maximize citation-worthy assets such as sourced syntheses, frameworks, decision matrices, checklists, calculators, templates, reference tables, datasets, glossaries, and original methodologies where appropriate;
- maximize valid machine readability, entity coherence, internal authority, distribution, indexation, source freshness, and feedback loops;
- use observed performance to compound page families and formats that actually earn surfacing;
- never guarantee an external citation/surfacing outcome.

## 6. KPI Truth Law

The following are distinct metrics and must never be conflated:

1. fanout opportunities;
2. admitted pages/content;
3. published pages;
4. indexed pages;
5. observed search visibility;
6. observed LLM surfacing/mentions;
7. observed external references/backlinks;
8. verified external citations.

Internal opportunities, generated pages, internal links, schema markup, or modeled citation potential are not citations.

A verified surfacing/citation event should retain evidence where available: source/provider, timestamp, query or prompt, surfaced/cited URL, and observation artifact.

## 7. Risk-Adjusted Overlay Law

The same scale architecture may apply across repos without forcing identical publication rates or identical runtime implementations.

- High-stakes legal, medical, financial, credit, compliance, or regulated content requires stricter source/claim gates and may publish much more slowly than low-risk planning content.
- Commerce/application runtime boundaries remain separate from authority content boundaries.
- Existing proven repo-specific engines should be extended, not replaced merely to make all repos look alike.

## 8. Validation and Artifact Law

Before a migrated authority-scale snapshot is delivered:

- verify ZIP integrity and packaging root;
- verify required fanout manifests/shards exist;
- verify the materialized runway count and shard hashes structurally;
- verify accepted-output freeze manifests/blobs exist and have no unscoped drift in the snapshot;
- verify repo-specific publication ceiling semantics are machine-readable;
- verify KPI truth contract exists;
- report any unproven runtime/provider layer honestly.

Full local validators, deploy checks, commit, and push remain the responsibility of the applicable local updater flow unless verification mode was explicitly requested.
